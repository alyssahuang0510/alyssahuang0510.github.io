[01]   Context背景

A search app that unifies local and cloud behind one query

一個把本機與雲端整合在單一查詢後的搜尋 App

File Search lets people query local and cloud files from one place, and ships pre-installed on ASUS Commercial notebooks, AiOs and desktops from 2026 onward. It isn’t a feature buried in a menu — it’s a peer app inside MyExpert, sibling to Chat, Knowledge Hub and ExpertMeet. Because it runs on ordinary commercial machines, it has to stay useful without a dedicated server and without a power user babysitting it.

File Search 讓使用者從單一入口查詢本機與雲端檔案，並自 2026 年起預載於 ASUS 商用筆電、AiO 與桌機。它不是藏在選單裡的功能，而是 MyExpert 內與 Chat、Knowledge Hub、ExpertMeet 並列的 App。正因為它跑在一般商用機器上，就必須在沒有專屬伺服器、也沒有專家盯著的情況下持續好用。

[03]   The problem under the UI介面底下的問題

The hard part wasn’t the search screen

難的不是搜尋畫面

File Search runs two indexes with very different costs. A light exact-match index can run continuously and is essentially free. A semantic index — the one that makes search feel intelligent — is far heavier: left unmanaged, it keeps consuming memory until the machine slows or stalls. On a commercial laptop that isn’t an edge case; it’s the default failure mode. So the real design problem was never the result list. It was this: how do you keep semantic search fresh without letting it degrade the device it runs on?

File Search 跑兩種成本天差地遠的索引。一個輕量的精準比對索引可以持續運作、幾乎沒有成本。而讓搜尋變聰明的語意索引重得多：若不加管理，它會不斷吃記憶體，直到機器變慢或卡死。在商用筆電上，這不是邊緣案例，而是預設會發生的失敗。所以真正的設計難題從來不是結果清單，而是：如何讓語意搜尋保持即時，又不拖垮它所運行的那台機器？

Exact-match index精準比對索引Continuous, low cost, needs no governance, no real risk.持續運作、成本低、不需治理、幾乎無風險。
Semantic index語意索引Periodic and heavy; high performance cost; needs active resource management; risk is memory exhaustion.週期性且沉重；效能成本高；需要主動的資源管理；風險是把記憶體吃滿。
[04]   The system I designed我設計的系統

A self-regulating index lifecycle

一套會自我調節的索引生命週期

I designed the semantic index to govern itself — a four-state lifecycle driven by live memory, disk headroom and the device’s power state.

我把語意索引設計成自我治理——一套由即時記憶體、磁碟餘量與電源狀態驅動的四狀態生命週期。

Fig. 1 — The semantic index governs itself: it builds while resources are sufficient, pauses as they tighten, prunes least-recently-used entries when pressure peaks, then settles once headroom returns.

圖 1 — 語意索引自我治理：資源充足時建立，緊縮時暫停，壓力達臨界時優先釋放最久未使用的項目，餘量恢復後停止釋放。

When resources are sufficient it builds; as they tighten it pauses; when pressure peaks it prunes, freeing the least-recently-used entries first; once headroom returns it settles and stops freeing. The one trade-off I refused to hide: a pruned entry briefly leaves semantic results — so recovery is ordered to bring the most-used files back first. The system never asks the user to manage any of this; it simply behaves.

資源充足時建立；緊縮時暫停；壓力達臨界時釋放，優先清掉最久未使用的項目；餘量恢復後停止釋放、回到穩定。我唯一拒絕藏起來的取捨：被釋放的項目會暫時離開語意結果——所以恢復時依使用頻率排序，先把最常用的檔帶回來。整套系統不要求使用者管理任何一環，它只是自己運作。

[05]   The conflict衝突

A conflict no single screen owns

一個沒有任何單一畫面會負責的衝突

The sharpest decision didn’t come from a layout. The first design gated indexing on a simple condition — the machine being idle and plugged in. But idle and sleep are two different timers, and a user can set sleep shorter than idle. When that happens, the machine sleeps before the idle threshold is ever reached, so indexing never runs and search silently goes stale.

最關鍵的決策不是來自排版。初版把索引綁在一個簡單條件上——機器閒置且插電。但閒置與睡眠是兩個不同的計時器，使用者可以把睡眠設得比閒置還短。一旦如此，機器會在到達閒置門檻前就先睡著，索引永遠不會執行，搜尋就悄悄地過期。

Fig. 2 — When the sleep timer is shorter than the idle threshold, idle never triggers — so the redesigned condition can’t strand the index.

圖 2 — 當睡眠時間短於閒置門檻，閒置永遠不會觸發——所以重新設計的條件不能把索引晾在那裡。

Why it matters: catching that timing conflict — and redesigning the trigger so it can’t strand the index — mattered more than any screen in the app. This is the kind of system-level conflict that belongs to no single feature, which is exactly why it’s easy to ship a product without it.

為什麼重要：抓出這個時序衝突、並重新設計觸發條件讓它無法把索引晾在那裡——這比 App 裡任何一個畫面都重要。這類系統級衝突不屬於任何單一功能，這也正是為什麼很容易做出一個少了它的產品。

[06]   Key decisions關鍵取捨

What I traded off — and why

我取捨了什麼——以及為什麼

  • 01

    Pruned least-recently-used first — and never hid the cost

    優先釋放最久未使用——且不隱藏代價

    Trade-off: freshness vs. honesty. Dropping an index frees memory but removes that file from semantic results, so I ordered both pruning and recovery by access recency — and made the behaviour predictable rather than silent.

    取捨：即時性，對上誠實。釋放索引能省記憶體，但也讓該檔離開語意結果，所以我讓釋放與恢復都依存取頻率排序——讓行為可預期，而非默默發生。

  • 02

    Two indexes, two policies — didn’t manage them the same way

    兩種索引，兩套政策——沒有用同一套管

    Trade-off: simplicity vs. fit. Managing both indexes identically would’ve been simpler — but the cheap one needs no governance and the expensive one needs all of it. Matching policy to cost kept the light index always-on and the heavy one safe.

    取捨：簡單，對上合身。用同一套方式管兩個索引比較省事，但便宜的那個根本不需要治理，貴的那個則需要全部。讓政策貼合成本，才能讓輕量索引永遠在線、沉重的那個保持安全。

  • 03

    Scoped as a peer app, not a buried feature

    定位為並列 App，而非被埋的功能

    Trade-off: discoverability vs. scope creep. I defended File Search as an app sibling to Chat and Knowledge Hub — not a search box inside something else — so it carries its own IA and can grow without being constrained by a host feature.

    取捨：可發現性，對上範圍蔓延。我守住 File Search 作為與 Chat、Knowledge Hub 並列的 App——而不是塞在別的東西裡的搜尋框——讓它擁有自己的資訊架構，能獨立成長、不被宿主功能綁住。

[07]   Outcome成果

It ships where most prototypes wouldn’t survive

它出貨在多數原型活不下來的地方

File Search ships pre-installed across ASUS’s 2026 commercial notebooks, AiOs and desktops, as a peer app inside MyExpert. The part I’m proudest of is the part users never see: the self-governing index is what lets semantic search run on ordinary, resource-constrained commercial machines — without a server behind it and without anyone tending it. The product-level decisions — peer-app scope, a clean result hierarchy, a defined MyExpert integration boundary — are what made it a product rather than a feature.

File Search 以 MyExpert 內的並列 App，預載於 ASUS 2026 商用筆電、AiO 與桌機。我最自豪的，是使用者永遠看不到的那一塊：自我治理的索引，讓語意搜尋能在一般、資源受限的商用機器上運作——背後沒有伺服器，也沒有任何人在照顧它。那些產品級決策——並列 App 的範圍、乾淨的結果階層、明確的 MyExpert 整合邊界——才讓它成為一個產品，而不是一個功能。

[08]   Reflection反思

In tool products, the best UX is often the part you never see

在工具型產品裡，最好的 UX 常常是你看不見的那塊

File Search reinforced what the TR7600 redesign first taught me: in genuinely complex tools, the highest-leverage design move is usually structural, not cosmetic. A polished result list means nothing if the index behind it stalls the machine, and the most important decision in the whole project belonged to no single screen. Designing the system — and the conflicts between systems — is the work.

File Search 印證了 TR7600 改版最早教會我的事：在真正複雜的工具裡，槓桿最大的設計動作通常是架構性的，而非表面的。如果背後的索引會卡死機器，再精緻的結果清單也沒有意義；而整個專案最重要的決策，不屬於任何一個畫面。設計系統——以及系統與系統之間的衝突——才是真正的工作。
