Redesigning the X-ray system that inspects every board before it ships

重新設計每片電路板出貨前都要過的 X 光檢測系統

Sole UI/UX ownerFull redesignQuantified impactSMT process engineers
唯一 UI/UX 負責人完整重新設計量化成效SMT 製程工程師
37%
less programming time on the main production board after the redesign (295→186 min), independently timed by FAE teams
改版後主要量產板的編程時間減少（295→186 分鐘），由 FAE 團隊獨立量測

TRI’s flagship 3D AXI software detects solder-joint defects on every PCB leaving a production line — operated in shifts, where a confusing flow costs real downtime. I led the full redesign from a legacy 2D AOI interface into a 3D-AXI-native workflow, then owned it for six years as the sole UI/UX designer. The redesign cut programming time by 37% on a 4,160-component board (295→186 min), independently measured by FAE teams — a denser 3,495-component board fell from 386→237 min too — with the heaviest gains in the Fine Tune and Library steps.

德律的旗艦 3D AXI 軟體負責檢測產線上每一片 PCB 的焊點缺陷——以輪班方式操作，一個令人困惑的流程就是真實的停機損失。我主導了從舊版 2D AOI 介面到 3D-AXI 原生工作流程的完整重新設計，並以唯一 UI/UX 設計師的身分負責了六年。改版讓 4,160 顆零件那塊板的編程時間減少 37%（295→186 分），由 FAE 團隊獨立量測——另一塊 3,495 顆零件的較密板也從 386→237 分——改善最大的是 Fine Tune 與 Library 兩個步驟。

  • Fine Tune −51% (158→77 min), Library −39% (124→76 min) — the two slowest steps. Collapsing F3/F4 into one Fine Tune page and replacing manual height calculation with mouse-wheel adjustment is what drove most of the saving.
  • Authored the UX Design Guideline — buttons, layout, dialogs, typography — and applied it as one standardized interface across the entire AXI line (SIII / LL / TL / F3D / FB / HP / SV families): a design system that survives every new machine.
  • Owned programming, alignment, GR&R, and multi-language deployment for global major accounts (automotive, server, semiconductor).
  • Worked directly with algorithm & mechanical teams; C# / MFC literacy made spec arguments shorter and feasibility calls faster.
  • Fine Tune −51%（158→77 分）、Library −39%（124→76 分）——兩個最慢的步驟。把 F3／F4 整併進單一 Fine Tune 頁、以滑鼠滾輪取代手動計算高度，是時間縮短的主因。
  • 撰寫了 UX 設計原則文件（按鈕、排版、彈窗、文字規範），並以此建立單一標準化介面、套用於整條 AXI 產品線（SIII / LL / TL / F3D / FB / HP / SV 系列）：一套每出一台新機都能延續的設計系統。
  • 獨力負責編程、定位、GR&R 與多國語系部署，服務汽車、伺服器、半導體等全球大客戶。
  • 直接與演算法與機構團隊協作；C# / MFC 的底子讓規格討論更短、可行性判斷更快。

Taking a standalone search app from zero onto ASUS’s 2026 commercial line

把一個獨立的搜尋 App 從零做起，預載進華碩 2026 年的商用機種

Product-level ownership0 → 1IA & competitive strategy
產品級 ownership0 → 1資訊架構與競品策略
1/ 5
sibling apps in MyExpert — File Search is an app, not a feature buried in a menu
MyExpert 中的並列 App——File Search 是一個 App，而非藏在選單裡的功能

File Search unifies local and cloud files behind a single query and pre-installs on ASUS commercial notebooks, AiOs and desktops from 2026 onward. I owned its UX from zero to launch: product positioning, competitive teardown, information architecture, result hierarchy, and the integration boundary with the rest of MyExpert (Chat, Knowledge Hub, ExpertMeet).

File Search 把本機與雲端檔案整合在同一個查詢底下，自 2026 年起預載於華碩的商用筆電、一體機與桌機。我從零到上線一手負責它的 UX：產品定位、競品拆解、資訊架構、搜尋結果排序，以及它和 MyExpert 其他模組（Chat、Knowledge Hub、ExpertMeet）之間的整合界線。

  • Reconciled two mental models in one result list — file-manager behaviour (browse, sort) and search behaviour (rank, snippet) — without making either feel bolted on.
  • Scoped it as a peer app to Chat and Knowledge Hub, defending product-level boundaries against feature-creep pressure.
  • Mapped the AI integration surface so File Search can graduate into AI workflows without re-architecting its IA.
  • Designed a self-regulating indexing lifecycle (index → pause → prune → settle) so the heavier semantic index frees memory before it can slow or crash the device — pruning least-recently-used entries first, with the trade-off made explicit.
  • 在同一份結果清單中調和兩種心智模型——檔案管理（瀏覽、排序）與搜尋（排名、摘要）——讓兩者都不像勉強拼裝。
  • 把它定位為與 Chat、Knowledge Hub 並列的 App，在功能擴張的壓力下守住產品級邊界。
  • 梳理 AI 整合介面，讓 File Search 日後能升級進 AI 工作流，而不必重構它的資訊架構。
  • 設計了一套會自我調節的索引生命週期（建立 → 暫停 → 釋放 → 穩定），讓較重的語意索引在拖慢或拖垮裝置之前主動釋放記憶體——優先釋放最久未使用的項目，並把取捨講清楚。

ASUS Config Manager extends Microsoft SCCM — adding BIOS configuration, faster updates, and remote management via Intune/DFCI. I inherited a queue of high-stakes features touching enterprise device fleets, where a confusing flow means real downtime on real machines.My job: make cryptographic and policy complexity legible to non-developer IT admins. This is security UX in practice — surfacing security posture and risk so admins act with confidence instead of fear.

ASUS Config Manager 擴充了微軟 SCCM——加入 BIOS 設定、加速更新，並透過 Intune/DFCI 進行遠端管理。我接手了一系列牽動整批企業裝置的高風險功能；在這裡，一個讓人搞不清楚的流程，就代表真實機器的停機。我的工作，是把密碼學和政策上的複雜，做成非技術背景的 IT 管理者也讀得懂、用得放心的介面。這就是真正落地的資安 UX——把安全狀態和風險清楚攤出來，讓管理者帶著信任、而不是恐懼去操作。

  • Encrypted BIOS deployment & Silent Update — surfaced security posture and update state without burying admins in jargon.
  • Driver Command Policy — designed version detection, severity filters, and customizable update cadence.
  • JSON config deployment & a consolidated result report — rewrote raw firmware exit codes into plain-language, item-level status, so a failed fleet deploy is legible at a glance instead of drilled out device-by-device.
  • 加密 BIOS 派送與靜默更新——把安全狀態與更新狀態清楚呈現，而不讓管理者淹沒在術語裡。
  • Driver Command Policy——設計版本偵測、嚴重度篩選與可自訂的更新頻率。
  • JSON 設定派送與整合結果報表——把原始韌體 exit code 改寫成逐項、白話的狀態，讓失敗的機隊部署一眼看懂，而不必逐台下鑽。

Designing platform features from zero — and resolving the conflicts no single feature owns

從零設計平台功能——也解決那些夾在功能之間、沒人負責的衝突

System-level UX0 → 1Cross-product
系統級 UX0 → 1跨產品線
6+
platform features designed from zero across two product lines
跨兩條產品線、從零設計的平台功能

When Android ships a platform capability, every OEM must design the UX above it — settings IA, entry points, toggle behaviour, and the conflicts between features that can’t coexist. I led from-zero UX for Predictive Back, Edge-to-edge, Circle to Search, Auto Rotate, default Gemini assistant, and gaming-phone gyroscope on ROG and Zenfone. The hardest work isn’t any single feature — it’s the system view.

Android 推出一項平台能力時，真正面對使用者的那層體驗，還是得每家 OEM 自己設計——設定怎麼編排、從哪裡進入、開關怎麼運作，還有那些無法並存的功能彼此之間的衝突。我在 ROG 與 Zenfone 上，從零主導了 Predictive Back、Edge-to-edge、Circle to Search、Auto Rotate、預設 Gemini 助理，以及電競手機陀螺儀的 UX。最難的從來不是某一個功能，而是看清楚這些功能怎麼湊成一個系統。

  • Conflict resolution — for mutually exclusive features, defined the rule set, the resolution UI, and the recovery path.
  • Cross-product differentiation — ROG (gaming, tunable) vs Zenfone (general); diverged the IA only where it earned its keep.
  • Also redesigned and stabilized Screen Recording after takeover.
  • 衝突處理——針對互斥功能，定義規則集、衝突解決的 UI 與復原路徑。
  • 跨產品線差異化——ROG（電競、可調校）對比 Zenfone（一般消費）；只在值得的地方讓資訊架構分流。
  • 另接手並重新設計、穩定了螢幕錄影功能。

[03]   How I work工作方法

Every decision sits on a traceable line — finding, decision, outcome.

每個設計決策都落在一條可追溯的線上——證據、決策、成果。

01 — Finding

Start from evidence

從證據出發

A usability study, FAE-measured data, a field complaint, a system constraint. I don’t move until the problem is named in numbers or observed behaviour.

一份可用性研究、FAE 量到的數據、一則現場客訴、一個系統限制。在問題能用數字或觀察到的行為說清楚之前，我不動手。

02 — Decision

Change one thing, on purpose

有意識地改一件事

Each design move ties back to a specific finding — and I can say what I traded off for it. Feasibility is part of the call; I read the engineering constraint, not just the mockup.

每個設計動作都連回某個具體發現——而且我說得出為此取捨了什麼。可行性是決策的一部分；我讀的是工程限制，不只是畫面。

03 — Outcome

Measure what moved

量出改變了什麼

Programming time, returns recovered, sources unified, SUS lift. If a change can’t be measured, I say so — and I don’t claim it.

編程時間、挽回的退貨、收斂的來源數、SUS 提升。如果一個改變無法被量測，我會直說——而且不會把它當成果宣稱。

Every case on this site is built this way. You can always trace a shipped outcome back to the finding that justified it — and that traceability is what I bring to a team, not just to my own work.

這個網站上的每個案例都是這樣寫的。你永遠可以把一個已落地的成果，回溯到當初支持它的那個發現——而我帶進團隊的，正是這種可追溯性，不只是我自己的產出。

[04]   About關於

A designer who wrote the code and owned the design — at the same time.

同時寫程式、也完全主導設計的設計師。

I’m a pure UX designer working on tool-shaped products — the kind operators, IT admins, and developers depend on daily. My undergraduate degree is in Information Management; my master’s is in Interaction Design. That dual root means I read engineering tickets the way I read research notes.

我是專注在工具型產品的純 UX 設計師——就是產線操作員、IT 管理者與開發者每天都得用的那種產品。我大學念資訊管理，研究所念互動設計。也因為有這兩種背景，我讀工程端的需求單，就跟讀使用者研究筆記一樣自然。

For six years I owned the UX of an industrial inspection product end-to-end while writing its C# / MFC code — engineering and design were one job, not two phases. My master’s thesis was an AR rehabilitation game in Unity. Code isn’t my main output anymore, but it shapes how I think about feasibility, system constraints, and exactly what to ask engineers for — which is why cross-functional teams move faster with me in the room.

在德律的那六年，我一邊寫 C# / MFC 程式、一邊完整主導這套工業檢測產品的 UX——工程與設計是同一份工作，不是先後兩個階段。碩士論文則用 Unity 做了一款 AR 復健遊戲。寫程式現在已經不是我的主要產出，但它讓我對可行性、系統限制、以及該跟工程師要什麼，都比一般設計師更有判斷——這也是為什麼，有我在的跨職能團隊溝通起來會更快。

The tools I gravitate toward are unglamorous: dense data, expert users, long task sessions, conflicting modes. My job is to make those legible — and increasingly, to build the tools my own design team plans with.

我喜歡做的工具都不華麗：資料密度高、使用者是專家、一用就是好幾個小時、各種模式還會互相打架。我的工作，就是把這些東西整理到讓人一看就懂、用得順手——而且這幾年越來越常是親手打造設計團隊自己拿來規劃的工具。
