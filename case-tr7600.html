[01]   Context背景

The software that inspects every board before it ships

每片電路板出貨前都要過一次的軟體

TRI’s flagship 3D AXI (Automated X-ray Inspection) software detects solder-joint defects on every PCB leaving an SMT production line. It runs in shifts, operated by process engineers under line-speed pressure — a confusing flow doesn’t just annoy, it stops production. The same software ships to global automotive, server, and semiconductor accounts, and must behave identically across more than ten machine variants.

德律的旗艦 3D AXI（自動 X 光檢測）軟體，負責檢測 SMT 產線上每一片 PCB 的焊點缺陷。它以輪班方式運作，由製程工程師在產線速度的壓力下操作——一個令人困惑的流程不只是惱人，而是會讓生產停擺。同一套軟體出貨給全球汽車、伺服器、半導體大客戶，而且必須在十餘種機種上行為一致。

[03]   Problem問題

One dense screen tried to be every stage at once

一個密集的畫面，硬要同時當每一個階段

The legacy interface stacked the entire inspection setup — CAD alignment, scan, fiducial, warpage, fine-tuning, indexing — onto a single canvas paired with one mega family-editor panel. Operators jumped between unrelated tasks without ever changing context. New engineers hit a wall: everything was visible, nothing was sequenced. Programming a single board ran as long as 386 minutes — over six hours — and stayed error-prone, because the screen never told you which stage you were in.

舊版介面把整個檢測設定——CAD 定位、掃描、Fiducial、翹曲、微調、索引——全部堆疊在單一畫布上，再配一個巨大的 Family editor 面板。操作員在毫不相關的任務之間跳來跳去，畫面卻從不切換情境。新進工程師撞牆：所有東西都看得到，卻沒有任何順序。為一片板編程最長要花上 386 分鐘——超過六小時——而且容易出錯，因為畫面從不告訴你現在身處哪個階段。

A note on what you’re looking at: this is a dense expert tool, not a consumer app. Visual density here is a constraint of the domain — process engineers need this much data on screen. The redesign goal was never to make it “pretty”; it was to make a complex workflow legible.

關於你看到的畫面：這是一套高密度的專家工具，不是消費型 App。這裡的視覺密度是領域的限制——製程工程師本來就需要這麼多資料在畫面上。改版的目標從來不是「變漂亮」，而是讓一個複雜的工作流程變得可讀。

[04]   Process過程

I matched the interface to the workflow, not the feature list

我讓介面對齊工作流程，而不是功能清單

I observed shift operators and programming engineers on real lines and found the bottleneck wasn’t styling — it was that one screen tried to be every stage at once. Timing the real programming flow also showed the clock was concentrated in two steps — Fine Tune and Library — so that’s where I aimed the heaviest redesign. So I mapped the actual inspection workflow into discrete stages, then designed a per-stage interface where each screen exposes only the controls and data that stage needs. The structural change, not the cosmetics, did the work.

我在真實產線上觀察輪班操作員與編程工程師，發現瓶頸不在視覺風格——而在於一個畫面硬要同時當每一個階段。為真實編程流程計時後也發現，時間集中在兩個步驟——Fine Tune 與 Library——所以我把最重的改版火力對準那裡。接著我把真實的檢測工作流程拆解成獨立階段，再為每個階段設計專屬介面，讓每個畫面只攤開該階段需要的控制項與資料。真正起作用的是結構性的改變，不是表面的修飾。

[06]   Key decisions關鍵取捨

What I traded off — and why

我取捨了什麼——以及為什麼

  • 01

    Split one screen into staged screens

    把單一畫面拆成分階段的畫面

    Trade-off: more navigation steps, but far less cognitive load per stage. For a shift tool used under time pressure, lower per-stage load wins — operators move faster when each screen has one clear job.

    取捨：多了切換步驟，但每個階段的認知負荷大幅降低。對一個在時間壓力下使用的輪班工具，降低單階段負荷更重要——當每個畫面只有一個明確任務，操作員反而更快。

  • 02

    Kept the dense data tables — didn’t hide them

    保留高密度資料表——而非把它藏起來

    Trade-off: a cleaner-looking screen vs. expert power. This is a tool for engineers who need the numbers; density is the constraint, not a flaw to design away.

    取捨：看起來更乾淨的畫面，對上專家的操作效率。這是給需要看數字的工程師用的工具；密度是限制，不是該被設計掉的缺陷。

  • 03

    One interface across all machine variants

    一套介面，套用所有機種

    Trade-off: per-machine optimisation vs. consistency at scale. I chose a single standardised interface so every new machine inherits a familiar flow — and so I could maintain it alone for six years.

    取捨：單機最佳化，對上規模化的一致性。我選擇單一標準化介面，讓每台新機都繼承熟悉的流程——也讓我能獨力維護它六年。

  • 04

    Rebuilt the Fine Tune step around the engineer’s hand

    圍繞工程師的手，重建 Fine Tune 步驟

    Trade-off: the legacy flow split fine-tuning across separate F3/F4 screens and made engineers calculate component height by hand. I merged them into one Fine Tune page and replaced the manual math with direct mouse-wheel adjustment — a denser single page, but far less to learn and no hand calculation. Result: the Fine Tune step dropped 51% (158→77 min) — the single biggest contributor to the overall saving, with Library close behind at −39%.

    取捨：舊流程把微調拆在 F3／F4 兩個畫面，還要工程師手動計算零件高度。我把它們整併成單一 Fine Tune 頁，並以直接的滑鼠滾輪調整取代手動計算——頁面更密，但要學的更少、也不必再手算。結果：Fine Tune 步驟減少 51%（158→77 分）——是整體時間縮短最大的單一貢獻，Library 緊接其後 −39%。

The redesigned interface shipped across the AXI product line to global automotive, server and semiconductor accounts, in multiple languages. FAE teams independently timed total programming on two production boards: a 4,160-component board fell from 295 to 186 minutes, and a 3,495-component board from 386 to 237 minutes — roughly a 38% reduction. The heaviest gains came from the two slowest steps — Fine Tune (−51%) and Library (−39%) — driven by collapsing F3/F4 into one page and replacing manual height calculation with mouse-wheel adjustment.

改版後的介面隨 AXI 產品線出貨給全球汽車、伺服器、半導體大客戶，並支援多國語系。FAE 團隊在兩塊量產板上獨立量測總編程時間：4,160 顆零件的板從 295 分降到 186 分，3,495 顆零件的板從 386 分降到 237 分——整體約減少 38%。改善最大的來自兩個最慢的步驟——Fine Tune（−51%）與 Library（−39%）——關鍵是把 F3／F4 整併成單一頁、以滑鼠滾輪調整取代手動計算高度。

[08]   Reflection反思

In industrial UX, structure beats styling

在工業 UX 裡，結構勝過修飾

The win here wasn’t making the software look modern — it was making a complex expert tool legible by matching the interface to the real workflow instead of the feature list. That lesson carried straight into my later work on enterprise IT tooling and platform UX: when the domain is genuinely complex, the highest-leverage design move is almost always architectural, not cosmetic.

這裡真正的成果，不是讓軟體看起來現代——而是讓一個複雜的專家工具變得可讀，靠的是讓介面對齊真實工作流程、而非功能清單。這個體會直接延續到我後來在企業 IT 工具與平台 UX 的工作：當領域真的很複雜，槓桿最大的設計動作幾乎總是架構性的，而不是表面的。
